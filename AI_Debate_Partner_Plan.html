AI Debate Partner

Pick any topic → the AI takes the opposite side and debates you in real time, scores your arguments, flags logical fallacies, and gives improvement tips.

1. Project Overview

What it is: An educational web app where a user chooses a debate topic and a side (Pro/Con). The AI argues the opposite side, and after each of the user's arguments it acts as a judge — producing a numeric score, detecting logical fallacies, and giving concrete tips to improve.

Why it stands out on a resume: It is real AI engineering (a dual-role LLM system with a structured scoring rubric), full-stack ownership (frontend + API + database + deploy), and a genuinely useful product angle beyond a plain chatbot.

Three things this project proves to a recruiter:
  1. Real AI integration — structured prompting, an LLM-as-judge scoring layer, JSON output parsing, and a provider-swappable architecture.
  2. Full-stack skill — React frontend, Node/Express API, PostgreSQL persistence, authentication, deployment.
  3. Product thinking — the argument-scoring + fallacy-detection feature is the differentiator. It is the headline talking point in interviews.

Key security rule: The Hugging Face token lives only on the backend in a .env file. The React app never sees it — it only talks to your own Express API. Exposing an API key in frontend code is an instant red flag to reviewers and a real security hole.

3. Choosing the Hugging Face Model

The app needs two things from the model: good reasoning (to argue and to judge fairly) and reliable JSON output (for the scoring). Bigger instruct-tuned models do both far better than tiny ones.

RoleRecommended modelWhy
Primary (debate + scoring)meta-llama/Llama-3.3-70B-InstructStrong reasoning, follows the "opposite side" persona and the scoring rubric well. Available via HF Inference Providers.
Cheaper / faster altQwen/Qwen2.5-7B-Instruct or mistralai/Mistral-7B-InstructFully free on the serverless tier, fast, fine for a demo. Weaker on nuanced scoring.
Local / offline (optional)Run Qwen2.5 via Ollama100% free, no rate limits, no token — needs a decent CPU/GPU.
Reality check on "free": HF's serverless Inference API is rate-limited and small models don't always guarantee valid JSON. Mitigation is built into the design (Section 6): the scorer request asks for strict JSON, and the backend has a JSON-repair + retry wrapper so a malformed score never crashes the app.

4. System Architecture & Logic Flow

┌──────────────┐ HTTPS/JSON ┌───────────────┐ HTTPS ┌──────────────────┐ │ React SPA │ ───────────────────────▶ │ Express API │ ──────────▶ │ Hugging Face │ │ (browser) │ ◀─────────────────────── │ (Node.js) │ ◀────────── │ Inference API │ │ │ │ │ └──────────────────┘ │ Web Speech │ token stays here ──▶ │ callLLM() │ │ API (voice) │ │ scoring svc │ └──────────────┘ └───────┬───────┘ │ Prisma ▼ ┌───────────────┐ │ PostgreSQL │ users · debates · turns · scores └───────────────┘

End-to-end flow of ONE debate turn

1. User types/speaks an argument in the React chat box. 2. Frontend POSTs { debateId, argument } to POST /api/debate/:id/turn (with JWT). 3. Express saves the user's Turn row in Postgres. 4. Express calls callLLM() TWICE, in parallel: (a) DEBATER → generates the AI's opposite-side rebuttal. (b) JUDGE → returns strict JSON: scores + fallacies + tips. 5. Backend parses/repairs the Judge JSON, saves an AI Turn + a Score row. 6. Backend responds: { aiRebuttal, score, fallacies, tips, runningTotal }. 7. Frontend renders the rebuttal in chat + a score card beside it. 8. (Voice mode) Web Speech API reads the AI rebuttal aloud.

Why two separate AI roles?

Mixing "argue against me" and "grade me fairly" in one prompt produces a biased judge. Splitting them into a Debater call and a Judge call keeps scoring objective and makes each prompt simpler and more reliable — this separation is itself a strong interview talking point.

6. The AI Logic (the heart of the app)

Role A — The Debater

System prompt locks the model to the assigned side, keeps rebuttals short (2–4 sentences), argues persuasively but fairly, and never concedes.

SYSTEM (Debater):
"You are debating the {aiSide} side of the topic: '{topic}'.
Argue persuasively for your side. Keep replies to 2-4 sentences.
Never concede or switch sides. Respond directly to the user's last point."

Role B — The Judge / Scorer

A second call that grades the user's argument and returns strict JSON. An explicit rubric (what a 1 vs a 10 means per category) keeps scores consistent.

SYSTEM (Judge):
"You are a debate coach. Score the user's argument on a 1-10 scale for:
 logic, evidence, persuasiveness, rebuttal.
 Detect logical fallacies (name + the exact quote + why).
 Give 1-3 concrete improvement tips.
 Respond with ONLY valid JSON in this exact shape:"

{
  "scores": { "logic": 7, "evidence": 5, "persuasiveness": 8, "rebuttal": 6 },
  "total": 26,
  "fallacies": [ { "name": "Straw Man", "quote": "...", "explanation": "..." } ],
  "tips": [ "Back your second claim with a concrete example." ]
}
JSON reliability: Because free HF models don't guarantee valid JSON, the backend wraps the Judge call in a helper that (1) extracts the first {...} block, (2) JSON.parsees it, and (3) if it fails, retries once with a "return valid JSON only" reminder before falling back to a safe default. This is in services/scoring.ts.

The swappable LLM layer (key design)

// server/src/services/llm.ts
export async function callLLM({ system, prompt, json }) {
  // Provider chosen by env var → HF today, Claude/local later, no app changes
  const res = await fetch(
    `https://api-inference.huggingface.co/models/${process.env.HF_MODEL}`,
    { method: "POST",
      headers: { Authorization: `Bearer ${process.env.HF_TOKEN}`,
                 "Content-Type": "application/json" },
      body: JSON.stringify({ inputs: buildChat(system, prompt),
                             parameters: { max_new_tokens: json ? 500 : 300,
                                           temperature: json ? 0.2 : 0.7,
                                           return_full_text: false } }) });
  const data = await res.json();
  return data[0].generated_text;
}

12. Watch Out For

  • API key security — HF token backend-only, never in the React bundle.
  • HF rate limits / cold starts — the first call to a serverless model can be slow ("model loading"). Show a loading state and retry after a short wait.
  • JSON reliability — always run the Judge output through the parse/repair helper; never trust raw model text.
  • Cost control — cap turns per debate and set max_new_tokens low.
  • Web Speech API support — best in Chrome. Note this in the README instead of fighting cross-browser gaps.
  • Scoring consistency — give the Judge an explicit 1-vs-10 rubric so scores are stable across turns.
